# ADR-006: Persisted Shopping Lists Instead of On-Demand Regeneration

**Status:** Accepted  
**Date:** July 2026  
**Deciders:** Engineering lead  
**Repository Path:** `/src/app/actions/shopping.ts`, `/src/lib/shopping-list.ts`, `/src/lib/shopping-list-persistence.ts`, `/supabase/migrations/009_shopping_list_metadata.sql`, `/supabase/migrations/010_shopping_demand_details.sql`

---

## Context

The shopping list in ShelfLife is computed from a formula:

```
Shopping Demand = Σ(meal plan ingredient requirements) − Pantry Supply
```

This computation involves:
1. Fetching the current meal plan and all planned meals' ingredients
2. Fetching the current pantry items
3. Resolving ingredients against the Food Knowledge Graph (`buildFoodResolver`)
4. Aggregating requirements across all meals (summing quantities for shared ingredients)
5. Subtracting pantry supply per ingredient (semantic matching)
6. Building a list of what to buy, with quantities

Early versions of ShelfLife **recomputed this on every page load**. The shopping page called a function that re-read the meal plan and pantry, re-ran the computation, and returned the result. This caused a cascade of correctness bugs.

### The v1 Bug

The `getShoppingList()` function originally called `regenerateShoppingList()` inside it, which:
1. Deleted all existing shopping list rows
2. Re-ran the full meal plan computation
3. Re-inserted new rows

When recipe cards called `getShoppingList()` to check what was already on the list (to avoid duplicates), it **inadvertently regenerated the entire weekly shopping list**. A click on "Add Missing Ingredients" for a single recipe replaced the whole shopping list with the full meal plan's computed requirements.

**Symptom observed in QA:** Clicking "Add Missing Ingredients" on a Spaghetti Bolognese recipe card added ~24 items marked "Needed for 3 meals", "Needed for 4 meals" — the full meal plan, not just the recipe's missing ingredients.

---

## Problem

What is the correct persistence model for the shopping list?

The tension is:
- **Freshness:** The shopping list should reflect the current pantry and meal plan
- **Correctness:** A read operation must not trigger a write
- **Flexibility:** Users can add individual recipe items, manual items, and full meal-plan items to the same list
- **Source tracking:** The UI should distinguish meal-plan items from manually-added items
- **Quantity awareness:** Each item must record its demand quantity, pantry supply quantity, and computed buy quantity

---

## Decision

**The shopping list is persisted to `shopping_list_items` and regenerated explicitly** — never implicitly on read.

### Separation of Read and Write Paths

```
Write path: regenerateShoppingList() / addMissingIngredientsToShoppingList()
                    ↓
        shopping_list_items (persisted)
                    ↓
Read path: getShoppingList() — reads shopping_list_items directly (no computation)
```

**`getShoppingList()`** is read-only. It executes a single `SELECT` against `shopping_list_items`. It never calls `regenerateShoppingList()`. It never calls `computeShoppingList()`.

**`regenerateShoppingList()`** is write-only. It:
1. Computes the full shopping demand via `computeShoppingList()`
2. Deletes all existing shopping_list_items for the user
3. Inserts new rows via `insertShoppingListRows()`

**`triggerShoppingListRegeneration()`** is the canonical trigger. It is called by:
- `addIngredient`, `updatePantryItem`, `deleteIngredient` (pantry mutations)
- `generateMealPlan`, `reorderMealPlanItems`, `replaceMealPlanItem` (planner mutations)
- `completeCookedMeal` (cooking completion)

**`addMissingIngredientsToShoppingList()`** is an additive write. It:
1. Fetches current pantry items
2. Fetches current shopping list items (read-only)
3. Runs `planMissingIngredientsForShoppingList()` to determine what to add
4. Inserts only the new items via `insertShoppingListRows()` (does not delete existing items, does not regenerate)

### The `source` Column

Every `shopping_list_items` row has a `source` column:

| Value | Meaning |
|---|---|
| `meal_plan` | Generated by `regenerateShoppingList()` from the meal plan |
| `manual` | Added by `addMissingIngredientsToShoppingList()` from a recipe card, meal card, or suggestion |

This allows the UI to distinguish items and display source information.

### Persisted Quantity Metadata (Migration 010)

```sql
ALTER TABLE public.shopping_list_items
  ADD COLUMN IF NOT EXISTS demand_quantity numeric,
  ADD COLUMN IF NOT EXISTS demand_unit text,
  ADD COLUMN IF NOT EXISTS pantry_quantity numeric,
  ADD COLUMN IF NOT EXISTS pantry_unit text,
  ADD COLUMN IF NOT EXISTS used_by_meals jsonb NOT NULL DEFAULT '[]'::jsonb;
```

These columns are populated at write time and read back by `getShoppingList()` for display. This avoids re-computation on read and ensures the UI shows exactly what was computed when the list was last regenerated.

### The `SHOPPING_LIST_SELECT` Constant

`src/lib/shopping-list-persistence.ts` exports:

```typescript
export const SHOPPING_LIST_SELECT =
  "id, ingredient_name, quantity, unit, category, checked, needed_for_meals, shortage_label, demand_quantity, demand_unit, pantry_quantity, pantry_unit, used_by_meals, source";
```

This is used by all callers of `select()` on `shopping_list_items` to ensure consistent column selection.

### The `insertShoppingListRows()` Function

All shopping list inserts go through this single function:

```typescript
export async function insertShoppingListRows(
  supabase: ShoppingPersistenceClient,
  rows: ShoppingListInsertRow[]
): Promise<ShoppingListRow[]>
```

**Callers:**
- `regenerateShoppingList()` in `src/app/actions/shopping.ts`
- `addMissingIngredientsToShoppingList()` in `src/app/actions/shopping-list.ts`

This is a canonical single-implementation requirement. No other code path may insert directly into `shopping_list_items`.

## Alternatives Considered

### Recompute on Every Page Load

The original v1 approach: `getShoppingList()` calls `regenerateShoppingList()` which recomputes and re-persists.

| Factor | Assessment |
|---|---|
| Freshness | Always fresh |
| Correctness | Read triggers write — any read causes a full regeneration |
| Bug surface | Any code calling `getShoppingList()` for duplicate checking triggers a full regen |
| "Add Missing" | Called `getShoppingList()` → triggered regen → replaced individual recipe additions with full meal plan |

Rejected: This was the v1 bug. A read must never trigger a write.

### In-Memory Computation, Never Persisted

Compute the shopping list server-side on every read, return it, never write to the database.

| Factor | Assessment |
|---|---|
| Freshness | Always fresh |
| Manual additions | Cannot persist manual item additions ("Add Missing" from recipe card) separately |
| Source tracking | Cannot distinguish meal-plan items from manually-added items |
| Checked state | Cannot persist checkbox state between requests |
| Quantity details | Must recompute demand/pantry breakdown on every render |
| Performance | Requires Knowledge Graph resolution on every page load |

Rejected: Cannot persist user state (checkboxes, manual additions). The shopping list is a user-owned document, not just a computed view. Manual items from recipe cards and meal-plan items must coexist.

### Event Sourcing (Shopping List as Event Log)

Record every "add item" and "remove item" event; derive the current list from the log.

| Factor | Assessment |
|---|---|
| Correctness | Very high — full audit trail |
| Complexity | Significant implementation complexity for a single-team consumer app |
| Merge conflicts | Multiple concurrent mutations (regen + manual add) must be reconciled |
| Query complexity | Deriving current state from event log requires aggregation on every read |

Rejected: Disproportionate complexity for the problem being solved. The persisted materialized view approach provides the necessary correctness guarantees with significantly less implementation overhead.

---

## Consequences

### Positive

- `getShoppingList()` is always a cheap `SELECT` — no computation, no Knowledge Graph resolution
- Manual additions (`addMissingIngredientsToShoppingList`) and meal-plan items coexist in the same table with clear `source` tracking
- Checkbox state persists between page loads and across devices (stored in database)
- Demand/supply quantities are computed once and stored — no re-computation on render
- The bug where reading the shopping list regenerated the full meal plan is architecturally impossible — read and write paths are separate functions

### Negative / Trade-offs

- The shopping list may become stale if the user modifies their pantry or meal plan without triggering a regeneration
- `triggerShoppingListRegeneration()` must be called in every mutation that affects shopping data — forgetting it leads to a stale list
- Full regeneration deletes all `meal_plan`-sourced items and re-inserts them — any UI state (expanded items, etc.) is reset

### Mitigations

- Every pantry mutation (`addIngredient`, `updatePantryItem`, `deleteIngredient`) calls `triggerShoppingListRegeneration()` as its final step
- Every planner mutation calls `triggerShoppingListRegeneration()` as its final step
- `completeCookedMeal()` calls `triggerShoppingListRegeneration()` after pantry deduction
- `manual`-sourced items are protected during regeneration — only `meal_plan`-sourced items are replaced

> **Note:** As of v2.0, full regeneration deletes *all* items (not just `meal_plan` items). Manual item protection across regeneration cycles is listed in [docs/product-roadmap.md](../product-roadmap.md#5-next--v21) as a v2.1 improvement.

## Future Implications

- When manual items are protected across regenerations (v2.1), the `source` column becomes the discriminator for which items survive a regen
- If a mobile app is added, the persisted shopping list is immediately available via a REST API — no recomputation required
- The `used_by_meals` JSONB column enables future features: "Remove all items for Spaghetti Bolognese from my list"

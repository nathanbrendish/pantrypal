## 2. PantryPal (Pre-ShelfLife)

**Commits:** `bf9617e` through `d6d759b` (scaffold and V2 feature batch)  
**Date:** 2026-07-11 to 2026-07-12  
**Package name at this time:** `pantrypal`  

### Summary

The application began as a `create-next-app` scaffold and was incrementally developed to a feature-complete v1 consumer kitchen app across one continuous build session.

### Major Features

**Authentication**
- Email/password registration and login using Supabase Auth + `@supabase/ssr`
- Session management via Next.js middleware
- Email confirmation callback at `/auth/callback`
- Password strength validation, live email validation
- Forgot password / reset password flow

**Pantry (MVP)**
- `pantry` table created in migration 001 (initial schema: `id`, `user_id`, `ingredient_name`, `created_at`)
- Basic add / delete / browse operations

**V2 Feature Batch (migration 002)**
- Extended `pantry` with `quantity`, `unit`, `expiry_date`, `updated_at`
- Created `meals_saved`, `meal_plans`, `meal_plan_items`, `shopping_list_items`
- Expiry tracking with status labels (expired / today / tomorrow / soon / ok)
- Recipe catalogue: 100 seeded recipes in `src/data/recipes.ts`
- AI meal suggestions via Google Gemini (`suggestMeals` action)
- Weekly meal planner generated by Gemini (3, 5, or 7 days)
- Deterministic shopping list generation (`computeShoppingList` in `src/lib/shopping-list.ts`)
- Receipt scanner: image upload → Gemini Vision OCR → ingredient review → pantry insert
- Global receipt drop zone (drag image anywhere in the app)
- Saved meals bookmarking

**Account Management**
- Settings page with profile, account, and about sections
- Update display name, update email (with Supabase confirm flow), change password
- Delete account via `delete_user_account()` RPC (migration 003)

### Database Changes

| Migration | Key Changes |
|---|---|
| 001 | Created `pantry` |
| 002 | Extended `pantry` (qty/unit/expiry); created `meals_saved`, `meal_plans`, `meal_plan_items`, `shopping_list_items` |
| 003 | `delete_user_account()` SECURITY DEFINER function |

### Known Issues at This Stage

- Shopping list was AI-generated prior to being made deterministic (fixed in V2 batch)
- `meal_plan_items.day_index` was being overwritten with sort position during drag-and-drop reorder (see Bug Fixes below)
- Planner first-click failed due to receipt-specific error string in `mapGeminiError` 
- Email confirmation broken in production due to missing `NEXT_PUBLIC_SITE_URL` and redirect allow-list

## 3. ShelfLife v1.0.0

**Commits:** `41e5b4b`, `f484897`, and subsequent polish commits through `e561f65`  
**Dates:** 2026-07-12

### Summary

Production-ready v1.0 consumer release. Feature-complete UI/UX overhaul, design token system, account lifecycle, and critical bug fixes. Application was subsequently rebranded from PantryPal to ShelfLife.

### Major Features

**UI/UX Overhaul**
- Central design token system: `src/styles/tokens.css` (spacing, radii, z-index), `src/styles/theme.css` (color/theme variables), `src/styles/base.css`
- `src/lib/design-system.ts` typed helper
- Semantic CSS tokens: `background / surface / card / border / text / muted / primary / success / warning / danger / expired / expiring / fresh / accent`
- Typography system: Display, H1–H3, Section Title, Body Large/Body, Small, Caption, Label, Button
- Animation keyframes: fade-in, slide-up/down, scale-in, check-pop; `prefers-reduced-motion` safe
- Dashboard as home screen with live stats, quick actions, top recipe recommendation, expiry warnings, week plan preview, shopping summary
- Mobile bottom navigation (`MobileBottomNav`)
- Design system components: `AppCard`, `Button` variants, `EmptyState`, `LoadingSkeleton`, `SearchBar`, `FilterBar`, `Dialog`, `ProgressBar`, `StatCard`, `QuickActionCard`

**Bug Fixes**
- Planner reorder: fixed `day_index` being overwritten with sort position — only `sort_order` now changes on reorder
- Planner first-click: `mapGeminiError` made context-aware (receipt vs meals vs planner)
- Email confirmation: `getSiteUrl()` helper added; `emailRedirectTo` now uses canonical origin; `/auth/callback` route exchanges code for session
- Receipt scanning: hardened against 503s with `withGeminiRetry` (3 attempts, 1s/2s/4s backoff)

**Developer Experience**
- `src/lib/gemini/config.ts` — configurable Gemini model via `GEMINI_MODEL` env; default `gemini-3.5-flash`
- `src/lib/gemini/retry.ts` — shared retry utility
- `src/lib/gemini/map-gemini-error.ts` — context-aware error mapping
- Deterministic shopping list (replaced AI generation): `src/lib/shopping-list.ts`

### Rebrand Commit (`e561f65`)

- Package name changed from `pantrypal` to `shelflife` in `package.json`
- All user-facing strings updated to "ShelfLife"
- Application URL: `https://myshelflife.co.uk`
- Metadata, OG tags, PWA manifest updated
- `src/lib/brand.ts` created with `APP_NAME`, `APP_DESCRIPTION`, `APP_URL`, `APP_SHORT_NAME`

### 4.2 Food Knowledge Graph V2

**Commit:** `9c8023d` — 2026-07-12  
**Migrations:** 005, 006, 007, 008

#### Summary

Replaced free-text food categories with a controlled vocabulary, introduced storage locations as a separate user-owned concept, added classification caching to the pantry, and built the semantic matching foundation.

#### 4.2.1 Controlled Food Taxonomy (Migration 005)

**New tables:** `food_categories`, `food_subcategories`, `storage_locations`

**17 seeded food categories:** Dairy, Fruit, Vegetables, Meat, Seafood, Bakery, Pasta & Rice, Herbs & Spices, Pantry Staples, Condiments & Sauces, Oils & Vinegars, Tinned & Jarred Foods, Frozen Foods, Snacks, Confectionery, Beverages, Baking

**"Unclassified" design decision:** Not a row in `food_categories`. Represented by `food_category_id = NULL` with `confidence_score < 40`. This prevents "Unclassified" from polluting analytics as a real category.

**6 seeded storage locations:** Fridge, Freezer, Cupboard, Pantry, Drinks Cabinet, Cellar

**Ownership principle:**
- Food classification is community-owned (via `community_foods`)
- Storage location is user-owned (`pantry.storage_location_id` is never overwritten by community data)

#### 4.2.2 ID-Based Community Classification (Migration 006)

Replaced free-text `primary_category`/`secondary_category` in `community_foods` with `food_category_id` / `food_subcategory_id` (FK references). Legacy text columns retained but no longer written.

**`classification_version` trigger:** `community_foods_classification_version` (BEFORE UPDATE) — monotonically increments `classification_version` only when classification-relevant fields change (`food_category_id`, `food_subcategory_id`, `confidence_score`, `status`).

**New/rewritten RPCs:**
- `community_food_tier(numeric)` — maps confidence score to tier string
- `community_food_category_agreement(uuid, uuid)` — % category vote agreement
- `refresh_community_food_aggregate(uuid)` — now resolves category/subcategory by ID majority vote
- `record_community_food_observation(...)` — now accepts `observed_food_category_id`, `observed_food_subcategory_id`, `observed_storage_location_id`
- `classify_community_food(...)` — explicit classification by ID
- `get_community_classifications(uuid[])` — returns NULL for category when confidence < 40
- `resolve_community_food(text)` — returns `suggested_storage_location_id` from vote mode

#### 4.2.3 Pantry Storage and Classification Cache (Migration 007)

Added to `pantry`:
- `storage_location_id` — user-owned FK to `storage_locations`
- `canonical_food_id` — FK to `community_foods` (the Knowledge Graph link)
- `cached_category_id`, `cached_subcategory_id` — cached classification for fast rendering
- `classification_version`, `classification_updated_at` — stale detection fields

**New RPC:** `refresh_stale_pantry_classifications()` — updates cached columns where `community_foods.classification_version > pantry.classification_version`. Called on every `/pantry` page load.

**`learnAndSnapshot` helper** (internal to `pantry.ts`): Non-blocking wrapper that calls community learning before every pantry write and returns a cache snapshot. Any failure returns an empty snapshot — community learning never blocks a pantry write.

#### 4.2.4 Semantic Matching Foundation (Migration 008)

Added `substitutable boolean` to `food_subcategories`. When `true`, all foods sharing that subcategory satisfy each other in recipe matching and shopping demand.

**Initial substitutable subcategories:**
- `Pasta & Rice` → `Pasta` (Macaroni, Penne, Spaghetti, Rigatoni, etc. are interchangeable)
- `Dairy` → `Cheese` (Cheddar, Mozzarella, Parmesan, Feta, Gouda, Brie are interchangeable)

**New batch RPC:** `resolve_community_foods(text[])` — resolves an array of ingredient names in one query. Used by `buildFoodResolver`.

**Seeded Knowledge Graph records (17):** Pasta, Macaroni, Penne, Rigatoni, Spaghetti, Fusilli, Tagliatelle, Linguine, Lasagne, Fettuccine, Cheese, Cheddar, Mozzarella, Parmesan, Feta, Gouda, Brie — all seeded as `status: 'verified'`, `confidence_score: 100`.

#### New Libraries

| File | Purpose |
|---|---|
| `src/lib/semantic-match.ts` | `foodsMatch()` and `matchesAnyPantry()` — semantic comparison primitive. Three tiers: canonical ID → substitutable family → string fallback |
| `src/lib/food-resolver.ts` | `buildFoodResolver()` — server-only. One RPC call + one subcategory query to build a complete `FoodResolver`. Returns `EMPTY_FOOD_RESOLVER` on error |
| `src/lib/pantry-stacking.ts` | `insertOrStackPantryItem()` — unified pantry INSERT. Checks for existing item with same canonical food + expiry + storage location + compatible unit; stacks (adds quantities) or inserts |
| `src/lib/food-classification.ts` | `CONFIDENCE_THRESHOLDS`, `tierFromConfidence()`, `tierLabel()`, `tierBadgeClasses()`, `isUnclassified()` |
| `src/lib/storage-locations.ts` | `suggestStorageLocationId()` — fallback map from category to storage |
| `src/types/taxonomy.ts` | `FoodCategory`, `FoodSubcategory`, `StorageLocation`, `ClassificationTier` types |

#### UI Changes

- **`AddIngredientForm`**: Refactored to use shared `IngredientFields` component
- **`IngredientFields`**: New shared component for consistent ingredient input across add form and receipt review. Calls `getCommunityFoodDefaults` on blur to prefill unit, expiry, and suggested storage location
- **`ReceiptIngredientReview`**: Refactored to use `IngredientFields` — same workflow and options as manual add
- **`PantryBrowser`**: Redesigned comfortable/compact view modes with structural distinction. Grouped by storage location
- **`RecipeCatalog`**: Uses `FoodResolver` for semantic recipe matching on the server

#### Architecture Changes

- `addIngredient` now calls `learnAndSnapshot` + `insertOrStackPantryItem`
- `saveScannedIngredients` now calls `enrichScannedItem` + `insertOrStackPantryItem` (sequential to enable batch stacking)
- Both paths converge on `insertOrStackPantryItem` — single insertion point for pantry

### 4.3 Architecture Consolidation

**Commit:** `3bed83f` — 2026-07-13 00:01

#### Summary

Investigated and fixed a bug where "Add Missing Ingredients" from the planner inserted the entire meal plan shopping list instead of recipe-specific items. Consolidation identified duplicate implementations and refactored everything to a single canonical path.

#### Root Cause

`getShoppingList()` was previously calling `regenerateShoppingList()`, which re-executed the entire weekly planner demand calculation. The `/shopping` page was triggering a full regen on every load.

#### Fix: Read-Only Shopping List

- `getShoppingList()` rewritten to be SELECT-only on `shopping_list_items`
- Shopping list persistence separated: metadata (`needed_for_meals`, `shortage_label`, `source`) is now stored at write time so reads never need to recompute
- Regression test added: verifies `getShoppingList` contains no imports from `shopping-list.ts` or planner functions

#### Migration 009: Shopping List Metadata

```sql
ALTER TABLE public.shopping_list_items
  ADD COLUMN IF NOT EXISTS needed_for_meals integer NOT NULL DEFAULT 1 CHECK (needed_for_meals >= 1),
  ADD COLUMN IF NOT EXISTS shortage_label text,
  ADD COLUMN IF NOT EXISTS source text NOT NULL DEFAULT 'manual' CHECK (source IN ('manual', 'meal_plan'));
```

#### Consolidated Add Missing Implementation

**Problem identified:** Multiple UI surfaces had independent implementations of "Add Missing Ingredients" producing inconsistent behaviour.

**Solution:**

| Component | Previous | After |
|---|---|---|
| `planMissingIngredientsForShoppingList` | Scattered | Extracted to pure `src/lib/add-missing-ingredients-core.mjs` |
| `addMissingIngredientsToShoppingList` | Partially shared | Sole server action; all UI routes through here |
| `MissingIngredientsSection` | Used inconsistently | Standardised entry point for all "Add Missing" UI |
| `AddMissingToShoppingButton` | Multiple callers | Always called via `MissingIngredientsSection` |
| `insertShoppingListRows` | Duplicated in two actions | Extracted to `src/lib/shopping-list-persistence.ts` |

**Verified callers (all route through `addMissingIngredientsToShoppingList`):**
- MealPlanner, RecipeCatalog, RecipeDetailModal, MealCard, SavedMealsList

#### Bug Fixed: "Fresh Basil" Not Inserted

**Root cause:** `categorizeIngredient()` classified "Fresh Basil" as `Herbs & Spices`, but `SHOPPING_CATEGORIES` in `src/types/v2.ts` did not include this category. The shopping trip UI had no section to render it in.

**Fix:** Added `"Herbs & Spices"` and `"Drinks"` to `SHOPPING_CATEGORIES` in `src/types/v2.ts`. Updated `SHOPPING_ICONS` in `shopping-trip.tsx`.

#### Semantic Matching Integration

- `addMissingIngredientsToShoppingList` now builds a `FoodResolver` before comparing against pantry
- `matchesAnyPantry()` used for pantry comparison — Macaroni in pantry satisfies Pasta requirement
- Unresolved ingredients (not in Knowledge Graph) fall back to string comparison and are never silently dropped

#### Regression Tests Added

`tests/recipe-shopping-list.test.mjs` expanded to cover:
- Recipe-level Add Missing does not invoke weekly shopping regeneration
- Shopping page read is read-only (no `computeShoppingList` import)
- Multiple missing ingredients: all inserted
- Semantic match: covered ingredient skipped
- Semantic non-match: ingredient inserted
- Unknown unresolved ingredient: falls back to display name, inserted
- Duplicate protection: repeated invocations do not create duplicates
- All UI entry points use shared `MissingIngredientsSection`
- Shopping UI renders every category emitted by `categorizeIngredient`
- All shopping insertions use one shared persistence path

### 4.4 Quantity-Aware Shopping and Cooking Confirmation

**Commit:** `4fec843` — 2026-07-13 00:28  
**Migrations:** 010, 011

#### Summary

Two major architectural enhancements implemented in one phase:
1. Shopping list replaced with a quantity-aware demand engine (`Shopping List = Total Demand − Pantry Supply`)
2. "Cook Meal" replaced with a confirmation workflow that deducts actual quantities used

#### 4.4.1 Quantity-Aware Shopping (Migration 010)

**New columns on `shopping_list_items`:**
```sql
ADD COLUMN IF NOT EXISTS demand_quantity numeric,
ADD COLUMN IF NOT EXISTS demand_unit text,
ADD COLUMN IF NOT EXISTS pantry_quantity numeric,
ADD COLUMN IF NOT EXISTS pantry_unit text,
ADD COLUMN IF NOT EXISTS used_by_meals jsonb NOT NULL DEFAULT '[]'::jsonb;
```

**Quantity parsing enhancements** (`src/lib/ingredient-match.ts`):
- `parseIngredientRequirement()` now handles leading multiplier (`"x3 Egg"`, `"3x Egg"`) and trailing multiplier (`"Egg x2"`)
- `knownUnits` expanded to include `cloves`, `tbsp`, `tsp`, `pcs`
- `unitsAreCompatible()` normalisation expanded

**`computeShoppingList` rewrite** (`src/lib/shopping-list.ts`):
- `ComputedShoppingEntry` now includes demand/pantry fields and `used_by_meals`
- `aggregateRequirements()` now groups semantically (by canonical ID or family) and sums quantities
- `pantrySupplyForRequirement()` replaces `findPantryMatch()` — aggregates total pantry supply in compatible units
- `collectMealRequirements()` parses quantities from `ingredients_used` strings

**Recipe quantity payloads:**
- `src/lib/recipe-shopping-ingredients.ts` — new helper `quantityAwareMissingIngredients(recipe, missingIngredients)` combines recipe ingredients with `recipe.typical_quantities`
- `RecipeCatalog` and `RecipeDetailModal` now pass quantity-aware strings (`"400g Spaghetti"`) to `MissingIngredientsSection` via `shoppingIngredients` prop

**`MissingIngredientsSection`** updated to accept optional `shoppingIngredients` prop (used as shopping payload while `ingredients` remains for display only)

**Shopping UI (`shopping-trip.tsx`) changes:**
- `quantityLabel()` displays `"Buy x3"` for count quantities, `"Buy 400 g"` for measured quantities
- `demandDetailLabel()` for expanded detail: "Need x5 / Have x2 / Buy x3"
- Items are expandable (toggle) showing demand, supply, and "Used by" meal list
- Category grouping made dynamic — no hardcoded category list; all categories emitted by `categorizeIngredient()` render automatically
- "Clear Shopping List" button added with `window.confirm` dialog

**`SHOPPING_CATEGORIES` and `ShoppingListItem` type** (`src/types/v2.ts`) extended with `demand_quantity`, `demand_unit`, `pantry_quantity`, `pantry_unit`, `used_by_meals`

#### 4.4.2 Canonical Pantry Consumption Engine (Migration 011)

**New table: `cooking_behavior_observations`** (anonymous INSERT-only):
```sql
CREATE TABLE IF NOT EXISTS public.cooking_behavior_observations (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  recipe_id text,
  recipe_name text NOT NULL,
  expected_ingredient text,
  actual_ingredient text NOT NULL,
  expected_quantity numeric,
  expected_unit text,
  actual_quantity numeric NOT NULL DEFAULT 0 CHECK (actual_quantity >= 0),
  actual_unit text,
  action text NOT NULL DEFAULT 'used' CHECK (action IN ('used', 'skipped', 'extra', 'substituted')),
  created_at timestamptz NOT NULL DEFAULT now()
);
```

**`src/lib/pantry-consumption.ts`** — new canonical consumption engine:
- `consumePantryForCookedMeal()` — FIFO deduction by expiry date, semantic food matching via `foodsMatch()`, unit compatibility via `unitsAreCompatible()`
- `deductIngredientFromPantry()` — internal; reduces quantity or deletes rows
- `observationAction()` — classifies each ingredient usage as `used` / `skipped` / `extra` / `substituted`

**`completeCookedMeal()` action** (`meals.ts`):
- New canonical action for cooking completion
- Accepts structured `CookingUsageInput[]` (expected vs actual for each ingredient)
- Calls `consumePantryForCookedMeal()` then `triggerShoppingListRegeneration()`
- Revalidates all relevant paths

**`cookMeal()` backward-compatibility wrapper:**
- Delegates to `completeCookedMeal()` with default quantities (`1`)
- Exists to avoid breaking existing callers

**`CookingConfirmationModal`** — new shared client component:
- Opens when user clicks "I've Cooked This Meal" on a recipe or meal card
- Defaults every ingredient to recipe quantities
- User can edit actual quantities, set to 0 to skip, add extra ingredients, or remove rows
- Submits to `completeCookedMeal()` on confirm
- **Integrated in:** `RecipeDetailModal`, `MealCard`

**Cooking workflow:**

```
User: "I've Cooked This Meal"
  ↓
CookingConfirmationModal opens
  ↓
User adjusts quantities (or confirms defaults)
  ↓
completeCookedMeal() called with actual usage
  ↓
consumePantryForCookedMeal()
  ├── Deduct quantities from pantry (FIFO by expiry)
  └── Record cooking_behavior_observations rows
  ↓
triggerShoppingListRegeneration()
  ↓
All affected paths revalidated
```

#### Regression Tests Added (migration 010/011 phase)

- `recipe Add Missing preserves recipe quantities in shopping rows`
- `recipe surfaces pass quantity-aware payloads to Add Missing`
- `shopping demand engine aggregates quantities and subtracts pantry supply`
- `shopping list supports clear all without planner or pantry mutation`
- `shopping rows persist quantity-aware demand details`
- `shopping insertions use one shared persistence path`
- `cooking completion uses canonical pantry consumption engine`
- `cooking confirmation supports exact, modified, skipped, and extra usage`
